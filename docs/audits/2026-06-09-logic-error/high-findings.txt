### [HIGH] loadRunMetadata unmarshals extract.json (mixed-type JSON) into map[string]string, always fails -> run detail page returns 404 for every run
- unit: gui | file: internal/gui/history.go:80-91 | cat: logic-error | conf(review/verify): high/high
- DESCRIPTION: loadRunMetadata reads extract.json and unmarshals into `var meta map[string]string`. It then returns nil if json.Unmarshal returns a non-nil error. - WHY BUG: buildRunDetail (page_handlers.go:118-121) does `meta := loadRunMetadata(...); if meta == nil { return nil }`, and RunDetailPageHandler (page_handlers.go:75-79) returns HTTP 404 "run not found" whenever buildRunDetail returns nil. I verified the unmarshal failure with a standalone Go program using the exact field set written by writeMetadataFile: err=`json: cannot unmarshal number into Go value of type string`. The correct approach is already used in internal/structure/extract.go, which unmarshals extract.json into a typed struct with only the `url` field.
- FIX: Unmarshal into map[string]any (or a typed struct mirroring writeMetadataFile) and coerce values to strings when building the metadata map, instead of map[string]string. E.g. decode into map[string]any, then for the fields the template needs (url, plan, version_string, edition, run_id) fmt.Sprintf the values into a map[string]string. Do not treat a non-nil error from a string-typed map as 'not found'.
- VERDICT: I attempted to refute this claim but the evidence confirms it. loadRunMetadata (history.go:80-91) unmarshals extract.json into `var meta map[string]string` and returns nil whenever json.Unmarshal returns a non-nil error. writeMetadataFile (extract.go:357-378) ALWAYS writes non-string values: `version` is a float (LegacyFloat, e.g. 2026.3), `plan` is [][]string, `target_tasks` and `available_configs` are []string.

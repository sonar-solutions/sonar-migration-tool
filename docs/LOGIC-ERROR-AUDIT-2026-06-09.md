# Logic-Error Hardening Audit — 2026-06-09
<!-- updated: 2026-06-09_21:16:18 -->

Exhaustive logic-error sweep of the entire Go codebase
(`go/cmd` + `go/internal`, ~42k non-generated lines across 183 files / 19
packages). Generated by a parallel agent workflow (`wf_471cf8cf-f9c`):
**34 reviewer agents** (one per coherent review unit, 100% non-generated
source coverage) → **adversarial verification** of every critical/high/medium
finding (refute-by-default) → synthesis. 88 agents, 6.7M tokens, ~42 min.

### HIGH-1 — Hotspot status sync never matches; every REVIEWED hotspot stays TO_REVIEW
- **File:** [go/internal/migrate/tasks_hotspotsync.go:270-324](../go/internal/migrate/tasks_hotspotsync.go#L270-L324), [#487-511](../go/internal/migrate/tasks_hotspotsync.go#L487-L511) · **category:** api-misuse · **confidence:** high (review) / high (verify)
- **Bug:** `findCloudHotspotCandidates` queries Cloud `/api/hotspots/search?files=…` and parses each result with `parseMatchableHotspot`, which reads `RuleKey` from `ruleKey`/`rule.key`. **SonarCloud's `/api/hotspots/search` does not return the rule key** (only `/api/hotspots/show` does), so every cloud candidate has `RuleKey == ""`. `classifyHotspotCandidatesByLine` then requires `c.RuleKey == sourceRule && c.Line == sourceLine`. Source hotspots carry a real rule (e.g. `python:S4823`), which can never equal `""`, so `matches == 0` → `syncOutcomeNotFound` → `syncOneHotspot` is never called → Cloud hotspots stay `TO_REVIEW`.
- **Why it's a bug / runtime evidence:** Matches the live 2026-06-09 run exactly: *"6 of 31 main hotspots lost REVIEWED/SAFE → TO_REVIEW (all python:S4823). Count migrated, status didn't sync."* The working **issue** path proves the asymmetry was understood: `findCloudIssueCandidates` passes `rules=ruleKey` (server-side scoping) and `classifyIssueCandidatesByLine` filters **by line only**. Hotspot sync took the opposite, broken approach.
- **Fix:** Don't match hotspots on a rule key from the search response. Match **by line only** (the search is already scoped to one file), exactly like the issue path. (Optionally call `/api/hotspots/show` per candidate to populate `RuleKey` if rule disambiguation is ever needed.)

### HIGH-2 — Per-branch resume is unreachable; a killed `importProjectData` silently drops un-migrated branches
- **File:** [go/internal/migrate/tasks_projectdata.go:44-49](../go/internal/migrate/tasks_projectdata.go#L44-L49), [#568-588](../go/internal/migrate/tasks_projectdata.go#L568-L588) + [migrate.go:382-387](../go/internal/migrate/migrate.go#L382-L387) · **category:** data-loss · **confidence:** high/high
- **Bug:** `runImportProjectData` builds a `completed` set via `loadCompletedBranches` and threads it through `shouldSkipBranch` for resume. But the whole task is gated upstream by `filterCompleted`, which drops **any** task whose output dir already exists — and that dir is created by `e.Store.Writer(...)` (`os.MkdirAll`) at the very *start* of the task, before a single branch is processed. So on any resume with the same `--run_id`, `filterCompleted` removes `importProjectData` (and downstream `syncIssueMetadata`/`syncHotspotMetadata`) from the plan, and the per-branch resume code never runs.
- **Why it's a bug:** Project with main + 5 long branches; main + 2 import, then SIGKILL/OOM/network drop. User resumes with the same run-id (the documented recovery path). `filterCompleted` sees the dir exists → drops the task → the 3 remaining branches are **silently absent on SonarCloud with no error**. The per-branch machinery is only exercised in isolated unit tests; nothing drives it through the `filterCompleted` gate.
- **Fix:** Make resumability branch-granular: exempt `importProjectData` from the dir-existence gate and rely on `loadCompletedBranches`/`shouldSkipBranch`, **or** only treat the task dir as complete when a sentinel/manifest records that every project+branch finished.

### HIGH-3 — Org-level group permission migration silently drops `profileadmin`/`gateadmin`/`provisioning`
- **File:** [go/internal/migrate/tasks_permissions.go:257-278](../go/internal/migrate/tasks_permissions.go#L257-L278) (filter at L265-269) + [helpers.go:296-300](../go/internal/migrate/helpers.go#L296-L300) · **category:** api-misuse · **confidence:** high/high
- **Bug:** `applyOrgPermissions` grants at **global/org** scope (`AddGroup(…, orgKey, "")`) but filters through `validPermissions`, which is the **project** permission set `{admin, codeviewer, issueadmin, securityhotspotadmin, scan, user}`. The valid global keys are `{admin, profileadmin, gateadmin, scan, provisioning}`. The overlap is only `{admin, scan}`, so a group holding global `profileadmin`/`gateadmin`/`provisioning` hits `!validPermissions[perm]` and is `continue`d **before any logging** — silently dropped.
- **Why it's a bug:** The adversarial verifier confirmed **SonarCloud's `api/permissions/add_group` at org scope accepts `profileadmin`/`gateadmin`/`provisioning`**, so this is genuine data loss, not a guard against an unsupported key. The sibling project-scoped path (`tasks_associate.go:274`) is correctly using the project set — which is exactly why copying that filter into the org path is wrong.
- **⚠️ Contradicts a documented decision.** [docs/CLOUDVOYAGER-DELTA.md](CLOUDVOYAGER-DELTA.md) states these org permissions are *intentionally* dropped because "Cloud doesn't support them." The verifier's API check says otherwise. **Confirm against the current SonarCloud API before fixing** — if Cloud now supports them, the doc is stale and this is a real regression.
- **Fix:** Add a separate `validOrgPermissions = {admin, profileadmin, gateadmin, scan, provisioning}` for the org path; keep `validPermissions` for the project path. Log when a permission is skipped so the loss is visible.

### HIGH-4 — `getAcceptedIssues` always uses the 10.4+ `issueStatuses` param, aborting extract on SonarQube Server < 10.4
- **File:** [go/internal/extract/tasks_issues.go:41-42](../go/internal/extract/tasks_issues.go#L41-L42) · **category:** api-misuse · **confidence:** high/high
- **Bug:** The task unconditionally queries `api/issues/search` with `issueStatuses=ACCEPTED,CONFIRMED`. That parameter and the `ACCEPTED` status value only exist from **SQ Server 10.4**; on 9.9–10.3 the param is `statuses` and `ACCEPTED` is a *resolution*. The rest of the codebase version-gates this exact param (`projectIssuesFullTask` switches on `issueStatusesRename` = 10.4; the `pipeline` package hardcodes legacy `statuses` for 9.9/10.0-10.3) — `getAcceptedIssues` is the one place missing the guard.
- **Why it's a bug:** On any 9.9–10.3 server (a common LTS band), the server returns HTTP **400**. `isNonFatalHTTPErr` only swallows 403/404, so a 400 propagates through `g.Wait()` and **aborts the entire extract phase** — the migration fails at step 1.
- **Fix:** Version-gate the param exactly like `projectIssuesFullTask`: `if e.Version.Less(issueStatusesRename)` use `statuses`, else `issueStatuses`. Optionally add 400 to the non-fatal set for best-effort count tasks.

### HIGH-5 — Edition `"Data Center"` (with a space) never matches the enum → silent downgrade to Community → DCE data skipped
- **File:** [go/internal/common/edition.go:60-65](../go/internal/common/edition.go#L60-L65) · **category:** incorrect-condition · **confidence:** medium (review) / high (verify, raised from medium)
- **Bug:** `parseEditionString` lowercases the raw value and matches exact literals `community/developer/enterprise/datacenter`. SonarQube's edition label for Data Center is **`"Data Center"`** (with a space, confirmed against `EditionProvider.java`), which lowercases to `"data center"` ≠ `"datacenter"`. The switch fails → `ParseEdition` falls through to its `EditionCommunity` default.
- **Why it's a bug:** With an admin token, `detectEdition` reads `api/system/info` first (the navigation fallback that returns lowercase `datacenter` is only hit on 403). The verifier parsed all 25 real `getServerInfo` captures in the repo and confirmed the edition lives at nested `System.Edition` in display-label form. Misdetected as Community, `FilterByEditionGeneric` strips every Enterprise/DCE-only task — `getPortfolios`, `getApplications`, `getProjectPullRequests`, `getUsage` — **silently, with no error and no user override** (the config `edition` field is "provisional, ignored").
- **Fix:** Normalise before matching: `strings.ReplaceAll(strings.ToLower(s), " ", "")` so `"Data Center" → "datacenter"`; and/or map known display labels. Map unrecognised non-empty values to a safe non-Community default or warn.

### HIGH-6 — Predictive report mis-classifies every project-level new-code-period
- **File:** [go/internal/predict/new_code_periods.go:79-122](../go/internal/predict/new_code_periods.go#L79-L122) · **category:** logic-error · **confidence:** high/high
- **Bug:** `synthesizeSetNewCodePeriods` treats a record as the project-level NCD only when `branchKey == ""`. But `api/new_code_periods/list` returns the project-level NCD against the project's **main branch name** (`"main"`/`"master"`), never empty. So the project-level branch is effectively dead code on real data, and main-branch records wrongly fall into the per-branch `!inherited` path that emits `ncd_branch_override=true`.
- **Why it's a bug:** The verifier confirmed against real fixtures: **0 records have `branchKey == ""`**. A project with a supported main-branch NCD (e.g. `PREVIOUS_VERSION`, no `inherited`) is falsely flagged as a per-branch override → the predictive report shows a **false Partial** with the wrong reason; real migrate applies it cleanly. The correct logic already exists in `runSetNewCodePeriods` (`branch == "" || branch == info.mainBranch`).
- **Fix:** Mirror migrate: index `createProjects` by `(server_url, key)` for `main_branch`, treat `branchKey == "" || branchKey == mainBranch` as project-level, and only flag `ncd_branch_override` for genuine non-main, non-inherited records. **Report-accuracy only — does not affect the actual migration.**

### HIGH-7 — Wizard `generateRunID` still uses the `count+1` algorithm that #361 fixed elsewhere → collides on gapped numbering
- **File:** [go/internal/wizard/helpers.go:100-110](../go/internal/wizard/helpers.go#L100-L110) · **category:** logic-error · **confidence:** high/high
- **Bug:** The wizard counts today's dirs and returns `today-(count+1)`. The sibling `migrate.generateRunID` and `extract.generateRunID` were deliberately changed to `max+1` (commit `3feb090`, "Fix runID collision when day's run-dir numbering has gaps", #361). The wizard's own comment says *"Mirrors migrate.generateRunID — keep them in sync,"* but #361 never touched it.
- **Why it's a bug:** With a gap in today's numbering (e.g. `-01`, `-03` after a deleted/aborted `-02`, or interleaved extract/migrate IDs), `count=2` → returns `-03`, colliding with the existing `-03`. The **extract path is fully exposed**: `phases.go` passes the wizard's ID verbatim as `cfg.ExtractID`, and `extract.go` only self-generates when `ExtractID == ""`, so it **silently clobbers/reuses a prior extract's output dir**. (The migrate path is masked because `RunMigrate` returns its own correct ID.)
- **Fix:** Replace with the `max+1` logic from the other two helpers (parse the numeric suffix, track the max, return `max+1`), or extract one shared helper so the "keep in sync" invariant actually holds.

### HIGH-8 — GUI run-detail page 404s for every real run (`extract.json` unmarshalled into `map[string]string`)
- **File:** [go/internal/gui/history.go:80-91](../go/internal/gui/history.go#L80-L91) · **category:** logic-error · **confidence:** high/high
- **Bug:** `loadRunMetadata` unmarshals `extract.json` into `var meta map[string]string` and returns `nil` on any unmarshal error. But `writeMetadataFile` writes mixed types — `version` is a float, `plan` is `[][]string`, `target_tasks`/`available_configs` are `[]string`. Unmarshalling those into a string-typed map always errors, so `loadRunMetadata` always returns `nil`.
- **Why it's a bug:** `buildRunDetail` returns `nil` when `meta == nil`, and `RunDetailPageHandler` then returns **HTTP 404 "run not found"** for every legitimate run. The verifier reproduced it with `go run` against a real production `extract.json`. Tests are green only because every stub writes a string-only `{"url": …}` object. The history *list* and the JSON API are unaffected (only clicking through to detail 404s).
- **Fix:** Unmarshal into `map[string]any` (or a typed struct like `structure/extract.go` already does) and coerce values to strings for the template; don't treat a string-map type error as "not found".
